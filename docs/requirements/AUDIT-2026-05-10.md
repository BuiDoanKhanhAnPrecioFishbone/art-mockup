# Mockup ↔ Requirements Sync Audit

> Snapshot taken 2026-05-10 against `docs/requirements/INDEX.md`.

## Doc-by-doc audit

### 01 — System Overview
- ✅ Sidebar groups (Assessment Management / Recruitment & Program / Template Library / Administration) match spec — `widgets/app-sidebar/model/navigation.ts`. Submissions/Candidates/Reports correctly hidden. Interview Criteria Template addition wired at `/templates/interview-criteria/page.tsx`.
- ⚠️ `app/(app)/candidates/page.tsx`, `reports/page.tsx`, `submissions/page.tsx` still exist as `PagePlaceholder` stubs — fine for now, but the routes are reachable.

### 02 — Recruitment Module
- ✅ Stage / Step / Step type model in `entities/program/model/workflow.ts` (`StepType = "default" | "interview" | "test"`); step settings (timeline, instruction, reviewerIds, emailTemplateId, scorecard) match.
- ⚠️ `MoveToStepModal` lets HR pick any stage/step manually; `app/api/cvs/[id]/promote/route.ts` stores whatever the client sends.
- ❌ **Auto-assignment algorithm absent** — only `autoAllocate: boolean` (round-robin) on `WorkflowStep`. No "fewest active candidates, tie-break by lower reviewer_id".
- ❌ "Always placed at first step of first stage on entry" not enforced — `/promote` accepts arbitrary stage/step.
- ❌ `Withdrawn`, `Completed` statuses + `Undo Reject` action absent. `entities/candidate/model/types.ts:9` only has `"on-going" | "hired" | "rejected"`.
- ❌ Step-type immutability after first candidate enters: not enforced.

### 03 — Program Setup
- ✅ Program Info tab (`widgets/program-form/ui/tabs/ProgramInfoTab.tsx`) covers name/headcount/period/job template/folder. Job-template snapshot copy at lines 42–77 has no write-back. Public Form: derived Status, Duration inherits from Hiring Period, General Information mandatory, iOS toggles for the rest. Workflow's "switching template replaces criteria" confirmation present.
- ⚠️ `validateForPublish` checks dates non-empty but **does NOT** enforce title min-3-chars, `endDate >= startDate`, or "job template must be Published" (job template has no `Status` field at all).
- ⚠️ Section ordering uses `kind` filters but doesn't hard-pin General/Skills to indices 0/1 in the data.
- ❌ Validation that `reviewer_ids` is non-empty when program is Active.
- ❌ Status `Closed` is offered in the dropdown — spec only allows `Draft | Active`.

### 04 — Email System
- ✅ Send-New-Emails dropdown (Send to Candidate / Send to Reviewer) wired with `initialReceiverType`. Two-column compose layout. Unified email log detail page adapts `ProgramEmail` → `EmailLog` shape on the fly. Skipped/Failed buckets surfaced; Resolve Issues modal extracts template vars + bounced-address fixes; transitions row to delivered.
- ⚠️ `EmailDelivery` only stores aggregate counts — no per-recipient `Skipped` record with issue reason. Log adapter synthesises placeholder skipped rows.
- ⚠️ Send-time validation (no email / not Active candidate) not implemented in `NewEmailPage.tsx`.
- ❌ Reply statuses (Accept / Decline / Reschedule) absent from `CandidateEmailReply`.

### 05 — Question Bank
- ✅ AI generation flow with HR review-before-save (`AIGenerateModal.tsx` → `/api/questions/ai-generate`). Code question payload has `testCases` with `visibleInTest` (≈ public/private).
- ⚠️ Question types diverge — implementation has `essay | multiple-choice | csharp | javascript | testing`; spec wants `Code | MultipleChoice | Essay | Rubric`. C#/JS split instead of unified Code; `Rubric` missing.
- ❌ `ai_generated_cases` flag on per-test-case entries.
- ❌ Type immutability after first use in a Published test.
- ❌ Title min-3-chars validation on POST.

### 06 — Test Management
- ✅ Static vs Dynamic composition mode in entity + `TestEditor.tsx`. Dynamic conditions with type/difficulty/tags/quantity. Pass Ratio + Allow Skip.
- ⚠️ Test status `"Draft" | "Published" | "Archived"` exists but no enforcement that "only Published tests" can be added to a step.
- ❌ Composition-mode immutability after first session.
- ❌ Step-type assignment validation ("Step type must be Test").
- ❌ Scoring rules per type — `Submission` only has flat `scorePercent`.

### 07 — Sessions
- ✅ Three session types in the type model (`SessionType = "Public" | "Private" | "Private Onsite"`).
- ❌ **Session status machine wrong** — `SessionStatus = "Active" | "Inactive" | "Closed"` instead of `Upcoming | Active | Closing | Completed | Cancelled`. `SessionsTab.tsx` filter even hard-codes Public/Private only.
- ❌ All status-transition rules, force-submit at end_time, Closing → Completed for Onsite, candidate add/remove with reasons, editing rules by session state.
- ❌ **Integrity monitoring entirely missing** — no `leaving_tab_count`, `copy_paste_count`, `devtools_open_count`, `multi_instance_count`, `multi_monitor_flag` fields. Zero hits across the repo.
- ❌ `force_submitted` flag and `actual_end_time` absent.
- ❌ Session-window-vs-test-duration validation in `SessionEditor.tsx`.

### 08 — Templates
- ✅ Section template entity has the 8 component types; General Information + Skills system sections protected. Criterion templates have 5-band `BehavioralGuideline`. Skill library has Pending Approval flow.
- ⚠️ `JobTemplate` lacks `Status` and the three skill tiers (must-have / nice-to-have / bonus). Tiering exists on `ProgramSkill.priority` but not at the source job template.
- ❌ min-2-options validation on radio/checkbox/dropdown components.
- ❌ Field name immutability + auto snake_case generation (only `label` editable).
- ❌ 409 conflict on duplicate skill across tiers.
- ❌ Scorecard locked when used by an active step.
- ❌ `Archived` status for criterion templates.

### 09 — UI Screens
- ✅ Test list columns. Session create form has Title / Test Template / Type / Refresh Access Code / Start / End.
- ⚠️ Session form lacks Program / Program Step / Candidates Selection (Private only); no Send Invitations toggle with sending-mode picker.
- ❌ Submission list, Submission Candidates view (HR/Reviewer), Individual Submission Detail (Skill Performance / Integrity Summary / AI Review / Final Review Result / Review full submission).
- ❌ All candidate-facing test-taking screens (Test Access / Overview / In Progress / Test Case Results).

### 10 — Data Integrity
- ✅ Program delete blocked while `status === "active"` (returns 409). Job-template snapshot pattern respected (no write-back).
- ❌ Soft-delete with `excluded=true + reason` not modelled on any entity.
- ❌ Status-change audit log (actor + timestamp + previous + new + reason) absent everywhere.
- ❌ Reason strings not required on Reject / Withdraw / Cancel.
- ⚠️ Scheduler/cron behaviour is out of scope for a pure mockup, but the data model could still capture `force_submitted` + `actual_end_time` for fidelity (currently doesn't).

### 11 — End-to-End Flow
- ✅ Stages 1–7 (job template → section templates → criteria → program create → candidate profile → public form → workflow): working screens linked from the sidebar; the flow holds together.
- ⚠️ Stages 8–9 (candidate entry, CV review): manual entry + CV bulk-upload work, but the auto-assign + first-step invariant is broken.
- ❌ Stages 10–12 (Test step, Interview step, Final decision) cannot be exercised end-to-end — no submission/session-detail UI, no test-taking flow, no integrity capture, no auto-promotion-on-pass logic.

## Top 10 gaps (prioritised)

1. **Auto-assignment algorithm absent.** Add a server-side helper that picks the reviewer with the fewest active candidates (tie-break by lower id) and forces entry at the first step of the first stage on every entry path (manual / SharePoint / public form).
2. **Session status machine wrong.** Replace `Active | Inactive | Closed` with `Upcoming | Active | Closing | Completed | Cancelled` and type-aware transitions. Without this the entire Submission/Sessions surface (Doc 07) is non-functional.
3. **Candidate status missing `Withdrawn` and `Completed` (terminal).** Update `CandidateStatus`, `ChangeStatusModal`, and add `Undo Reject` (HR-only).
4. **Submission / candidate test-taking flow does not exist.** No Submission List, no Submission Candidates view, no Individual Submission Detail, no candidate-facing Test Access / Overview / In Progress / Test Case Results. Blocks Doc 06 / 07 / 09 / 11 end-to-end demoability.
5. **Integrity monitoring absent.** No flags on Submission and no UI for `Undetected / Cheating`. Seed mock values so the Submission Detail view is realistic.
6. **Composition Mode / Step Type / Question Type immutability not enforced.** Add server-side checks in `tests/[id]`, program PATCH, and `questions/[id]` that reject changes once the linked entity has progressed.
7. **Job template missing Status + skill tiers.** Add `status: "Draft" | "Published" | "Archived"` and tiered skills. Then enforce "Job template must be Published" in Program Info.
8. **Reply status (Accept / Decline / Reschedule) and per-recipient Skipped record.** Add reply status + send-time validation that materialises a real `Skipped` recipient with `issueReason` instead of synthesising it in the log adapter.
9. **No status-change audit trail / soft-delete.** Add `statusHistory: StatusEvent[]` to Program / Candidate / Session and an `excluded?: { at, by, reason }` field on Candidate.
10. **Section template radio/checkbox/dropdown min-2-options + immutable snake_case `name`.** Validate option count in the editor and split immutable `name` from editable `label` per Doc 08.
